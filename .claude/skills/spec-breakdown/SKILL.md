# 需求拆解

将以下需求拆解为可执行的任务清单：

**需求输入：** $ARGUMENTS

---

## 执行步骤

### 1. 上下文加载

读取项目文档，建立需求边界认知：

- `docs/SPEC.md` — 技术规格（SDK 接口、API 契约、数据模型）
- `docs/ARCHITECTURE.md` — 系统架构（服务拓扑、数据流）
- `docs/DESIGN.md` — 技术设计（选型、模式、约定）
- `docs/tasks/CURRENT.md` — 当前任务状态

### 2. 需求澄清

逐项检查，如有不明确的点**必须向用户提问**，不得假设：

- [ ] 影响哪些服务？（Gateway / Processor / Dashboard 等）
- [ ] 核心场景是什么？（用一句话描述）
- [ ] 成功标准是什么？（怎样算"做完了"？）
- [ ] 与现有功能的关系？（扩展 / 独立 / 替代）
- [ ] 是否涉及新的基础设施或外部依赖？

### 3. 用户场景拆解

按 User Story 格式输出：

```markdown
### US-XX: {场景名称}

**作为** {角色}，**我想** {目标}，**以便** {价值}。

**验收标准：**
1. {可验证的条件}
2. {可验证的条件}

**涉及服务：** {Gateway / Error Processor / Dashboard 等}
```

### 4. 任务拆解

将每个场景拆解为开发任务，每个任务满足：

- **原子性** — 单个任务可独立完成和验证
- **可估量** — 明确的输入、输出、涉及文件
- **有序性** — 标注依赖关系和建议执行顺序

```markdown
### T1: {任务名称}
- 涉及包: `apps/xxx` 或 `packages/xxx`
- 依赖: 无 / T2
- 验收: {怎样算完成}
```

### 5. 影响评估

| 维度 | 评估 |
|------|------|
| 新增文件 | 列出预计新增的文件路径 |
| 修改文件 | 列出需要修改的现有文件 |
| 架构影响 | 是否需要新增服务 / 队列 / 数据表 |
| 文档更新 | SPEC / ARCHITECTURE / DESIGN 是否需要更新 |

### 6. 输出产物

1. **用户场景列表**（US-XX 格式）
2. **任务清单**（T1-TN 格式，含依赖关系）
3. **影响评估表**
4. **建议的 `docs/tasks/CURRENT.md` 更新内容**

> 注意：本 skill 仅做分析和拆解，不执行代码编写。

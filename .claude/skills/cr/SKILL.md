# 代码审查（CR）

对当前变更执行量化评分审查，输出六维打分报告和分级问题清单。Critical 归零方可合并。

> 本 Skill 可独立使用或作为 `/feat` 的 Phase 6 执行。详细评分量规、校准示例、反模式清单见 [`../feat/references/review-rubric.md`](../feat/references/review-rubric.md)。

## 审查范围

```!
git diff --stat HEAD
```

```!
git status --short
```

**指定范围：** $ARGUMENTS

---

## 执行流程

### 1. 加载审查标准

读取项目规则文件：

- `.claude/rules/review.md` — 审查 Checklist
- `.claude/rules/architecture.md` — 架构红线
- `.claude/rules/coding.md` — 代码规范

### 2. 变更理解

读取所有变更文件的完整内容：

- 逐文件阅读 diff，理解**做了什么**和**为什么**
- 识别变更类型：新增功能 / 重构 / 修复 / 配置变更
- 标注影响范围：仅限包内 / 跨包 / 跨服务

### 3. 六维评分

| 维度 | 权重 | 核心检查点 |
|------|------|-----------|
| 正确性 | 25 | 逻辑缺陷、边界处理、异常传播、幂等性 |
| 可读性 | 20 | 命名自解释、控制流扁平、函数 ≤30 行 |
| 可维护性 | 20 | 单一职责、依赖方向、无重复代码 |
| 安全性 | 15 | 输入校验、密钥管理、权限控制 |
| 性能 | 10 | N+1 查询、并行化、SDK 体积预算 |
| 测试覆盖 | 10 | AC 覆盖完整性、边界用例、测试文件位置 |

> 每维度的分数段标准、检查要点、扣分参考见 [`review-rubric.md`](../feat/references/review-rubric.md)。

### 4. 严重度分级

```
发现问题
├─ Bug / 安全漏洞 / 数据损坏 / 架构红线违反？→ 🔴 Critical（阻塞合并）
├─ 影响可维护性/性能/扩展性？→ 🟠 Major（强烈建议）
├─ 代码质量可后续改进？→ 🟡 Minor（建议）
└─ 风格/习惯问题？→ 🔵 Nitpick（可选）
```

**速查**：`any`/空 catch/apps 间 import/硬编码密钥/SQL 注入/SDK 引 Node API → 🔴；队列名硬编码/N+1/手写类型代替 `z.infer<>`/测试文件在 `src/` → 🟠

### 5. 质量门

| 总分 | 评级 | 合并条件 |
|------|------|----------|
| 90–100 | ✅ 优秀 | 可直接合并 |
| 75–89 | 🟡 良好 | 解决 Minor 后合并 |
| 60–74 | 🟠 一般 | 解决 Major 后合并 |
| < 60 | 🔴 较差 | 建议重构后重新 Review |

**特殊规则**：任意 Critical 未解决，无论总分多少，**不得合并**。

**变更概述**：[一句话描述变更内容]

### 综合评分

|------|------|------|------|

| 可维护性 | X | 20 | ... |
| 安全性   | X | 15 | ... |

### 🔴 Critical（必须修改，阻断合并）

**[CR-C01]** `path/to/file.ts:行号`
- **问题**：[一句话描述]
- **影响**：[具体后果]
- **修改建议**：
  ```language

  [原代码]

  // After
  [改后代码]
  ```

### 🟡 Minor（建议）
[同上格式]

### 🔵 Nitpick（可选）

---

### 亮点（不可省略，至少一条）
- ✨ [真实优点，不要敷衍]

### 行动项
**Blocker（合并前必须解决）**
- [ ] [CR-C01] ...

**Non-blocker（可后续迭代）**
- [ ] [CR-M01] ...
```

---

## 约束

- 每个扣分项**必须附代码证据和 Before/After 示例**，不接受泛泛描述
- 亮点不可省略，至少一条真实的优点
- 本 Skill 仅做审查，不自动修改代码。如需修复，由用户确认后手动执行
- 作为 `/feat` Phase 6 执行时，产出格式与此相同，CR 编号纳入可追溯 ID 链

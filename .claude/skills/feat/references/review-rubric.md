# Code Review 评分细则

> 供 Phase 6 使用。定义六维评分的详细量规、严重度分级决策树、常见问题模式库。

---

## 一、六维评分量规

### 1. 正确性（25 分）

| 分数段 | 标准 |
|--------|------|
| 23-25 | 逻辑无缺陷，所有边界条件正确处理，异常路径有完整覆盖，类型系统充分利用 |
| 18-22 | 主路径正确，大部分边界已处理，存在个别未覆盖的异常分支 |
| 12-17 | 存在可导致特定场景下错误输出的逻辑漏洞，或关键边界未处理 |
| 0-11 | 主路径有逻辑错误，或存在可导致数据损坏/安全问题的缺陷 |

**检查要点**：
- 条件分支是否完整（缺 else / default）
- null/undefined 是否在每个链路节点都有处理
- 异步操作的错误是否被正确传播（非空 catch）
- 幂等性：重复执行是否产生副作用
- 数值计算是否有溢出/精度丢失
- 队列消费：失败重试是否正确、是否可能死循环

### 2. 可读性（20 分）

| 分数段 | 标准 |
|--------|------|
| 18-20 | 命名自解释，代码块职责清晰，控制流扁平，无需注释即可理解 |
| 14-17 | 大部分代码清晰，部分变量名偏简或函数略长（>30 行） |
| 8-13 | 存在单字母变量、缩写词、嵌套 >3 层的控制流、魔法数字 |
| 0-7 | 命名误导，控制流混乱，大量注释解释"做什么"而非"为什么" |

**检查要点**：
- 变量/函数名是否自解释（避免 `data`、`tmp`、`val`）
- 函数是否过长（>30 行需审视）
- 嵌套层级是否 >3
- 魔法数字/字符串是否提取为常量
- 注释是否说明"为什么"而非"做什么"
- 中文注释是否符合项目规范

### 3. 可维护性（20 分）

| 分数段 | 标准 |
|--------|------|
| 18-20 | 单一职责到位，接口设计清晰，修改一个能力只需改一处，依赖方向正确 |
| 14-17 | 基本遵循 SOLID，个别函数职责略广或存在少量重复代码 |
| 8-13 | 存在明显重复代码，模块间耦合过紧，或接口设计不自然 |
| 0-7 | 严重违反单一职责，修改一处需联动多处，依赖方向错误 |

**检查要点**：
- 是否存在跨模块直接导入 Controller（应导入 Service）
- apps 之间是否有直接引用（应通过 shared 或队列）
- 队列名是否硬编码（应定义在 packages/shared）
- 是否存在重复的 Zod Schema 或手写类型（应从 Schema derive）
- 新增模块是否遵循 NestJS 模块结构规范
- 是否存在循环依赖

### 4. 安全性（15 分）

| 分数段 | 标准 |
|--------|------|
| 13-15 | 输入校验完整，无密钥泄露，权限控制正确，安全头配置合理 |
| 10-12 | 核心安全措施到位，个别边缘校验可加强 |
| 6-9 | 存在输入校验缺口，或敏感信息可能被日志输出 |
| 0-5 | 存在可被利用的安全漏洞（注入、越权、密钥硬编码） |

**检查要点**：
- 用户输入是否有 Zod Schema 校验
- 是否存在硬编码密钥 / Token / API Key
- 环境变量是否通过 Zod 校验而非直接 `process.env`
- SQL/NoSQL 查询是否防注入
- 错误响应是否泄露内部实现细节
- DSN publicKey 是否包含 secret 信息

### 5. 性能（10 分）

| 分数段 | 标准 |
|--------|------|
| 9-10 | 算法复杂度合理，无 N+1 查询，关键路径有缓存策略 |
| 6-8 | 主路径性能良好，存在个别可优化的次要路径 |
| 3-5 | 存在 N+1 查询、不必要的循环、或未设置超时 |
| 0-2 | 存在 O(n²) 及以上的热路径算法或明显资源泄漏 |

**检查要点**：
- 是否存在 N+1 查询（循环内 DB 调用）
- 是否有不必要的 await（可并行的串行化）
- 大数据量是否分页/流式处理
- 队列消费者是否有合理的并发配置
- SDK 体积是否在预算内（gzip ≤ 15KB）

### 6. 测试覆盖（10 分）

| 分数段 | 标准 |
|--------|------|
| 9-10 | 核心逻辑有单元测试，关键集成链路有覆盖，边界和异常用例齐全，AC 全部覆盖 |
| 6-8 | 主要 Happy Path 有覆盖，边界用例不完整，AC 大部分覆盖 |
| 3-5 | 只有少量测试，或测试不验证行为而是验证实现细节 |
| 0-2 | 无测试，或测试全部通过 mock 绕开核心逻辑 |

**检查要点**：
- 测试文件是否位于 `tests/`（非 `src/`）
- 集成测试是否使用真实 Dockerized PG（禁止 mock 数据库）
- 每条验收标准是否有对应测试用例
- 测试是否验证行为而非实现细节
- SDK 测试是否覆盖浏览器环境

## 二、严重度分级决策树

```
发现一个问题
  ├─ 会导致 Bug / 安全漏洞 / 数据损坏 / 架构红线违反？
  │   └─ YES → 🔴 Critical（必须修改，阻塞合并）
  │
  ├─ 会影响可维护性 / 性能 / 扩展性？
  │   └─ YES → 🟠 Major（强烈建议修改，非阻塞）
  │
  ├─ 影响代码质量但可后续改进？
  │   └─ YES → 🟡 Minor（建议改进）
  │
  └─ 风格/习惯问题，纯主观偏好？
      └─ YES → 🔵 Nitpick（可选）
```

### 严重度判定速查

| 典型问题 | 严重度 |
|----------|--------|
| `any` / `@ts-ignore` 绕过类型系统 | 🔴 Critical |
| 空 catch 块吞掉异常 | 🔴 Critical |
| apps 间直接 import（绕过架构红线） | 🔴 Critical |
| 硬编码密钥 / Token | 🔴 Critical |
| SQL 注入 | 🔴 Critical |
| SDK 中使用 Node.js API（破坏浏览器兼容） | 🔴 Critical |
| 队列名硬编码在 Worker 中 | 🟠 Major |
| 函数 >50 行，职责混杂 | 🟠 Major |
| N+1 查询 | 🟠 Major |
| 手写类型代替 Zod `z.infer<>` | 🟠 Major |
| 测试文件位于 `src/` 下 | 🟠 Major |
| 注释使用英文而非中文 | 🟡 Minor |
| 魔法数字未提取常量 | 🟡 Minor |
| 变量名偏简（如 `d`、`v`） | 🟡 Minor |
| 缺少 Swagger 装饰器 | 🟡 Minor |
| 未使用的 import | 🔵 Nitpick |
| `let` 可改为 `const` | 🔵 Nitpick |
| 代码格式轻微偏差（Prettier 可自动修复） | 🔵 Nitpick |

## 三、评分校准示例

### 场景 A：新增 NestJS 模块（标准交付）

| 维度 | 得分 | 评语 |
|------|------|------|
| 正确性 | 23 | Module/Controller/Service 结构规范，Zod 校验完整，异常有 AppException 包装 |
| 可读性 | 19 | 命名符合项目规范，函数职责单一，控制流清晰 |
| 可维护性 | 18 | 模块通过 DI 注入，队列名在 shared 中定义，无重复代码 |
| 安全性 | 13 | 输入校验到位，无密钥泄露 |
| 性能 | 9 | 无明显性能问题 |
| 测试覆盖 | 8 | 有单元测试覆盖核心逻辑，集成测试待补 |
| **总分** | **90** | ✅ 优秀 |

### 场景 B：SDK 新增插件（快速迭代）

| 维度 | 得分 | 评语 |
|------|------|------|
| 正确性 | 20 | 主路径正确，IntersectionObserver 兼容性处理有遗漏 |
| 可读性 | 17 | 函数略长（42 行），变量命名可优化 |
| 可维护性 | 16 | 插件模式符合现有规范，但与 Breadcrumbs 插件存在重复初始化逻辑 |
| 安全性 | 14 | 浏览器环境，无密钥泄露风险 |
| 性能 | 8 | 未做 throttle，高频滚动场景可能触发过多回调 |
| 测试覆盖 | 7 | 有基本浏览器测试，边界和异常缺失 |
| **总分** | **82** | 🟡 良好 |

## 四、常见反模式清单

| 反模式 | 严重度 | 修复方向 |
|--------|--------|----------|
| `catch (e) { }` 空捕获 | 🔴 | 至少记录日志或重新抛出 |
| `as unknown as T` 双重断言 | 🔴 | 使用 Zod parse 或 type guard |
| 循环内 `await` 逐个执行 | 🟠 | `Promise.all` 并行化 |
| 环境变量拼接过 URL | 🔴 | 使用 Zod 校验后 URL 构造 |
| Controller 中直接写业务逻辑 | 🟠 | 移至 Service 层 |
| 模块间通过 `forwardRef` 解决循环依赖 | 🟠 | 重新设计模块边界消除循环 |
| `setTimeout` 硬编码等待（非测试） | 🟠 | 使用事件驱动或 Poll 机制 |
